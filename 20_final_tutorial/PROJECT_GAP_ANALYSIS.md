## 下一批高价值优化

### P1：雷达领域闭环

理由：雷达搜索跟踪与当前 EOIR 传感器链路最接近，最适合作为第二类真实想定。

完成定义：

- 真实 AFSIM radar demo 来源；
- 雷达平台/传感器/事件组件；
- 雷达 IR 与领域约束；
- 搜索、发现、建航、丢航指标；
- 正向、边界和拒绝样例；
- 参数扫描与人工解释。

### P1：生产核心模块化

当前 01-07 步为了教学保留了代码演进和部分重复。生产入口不应继续复制。

目标结构：

```text
afsim_agent_core/
  config.py
  ir/
  components/
  retrieval/
  generation/
  validation/
  rendering/
  simulation/
  evidence/
  experiments/
```

迁移时保留旧步骤作为课程快照，新生产入口只依赖核心包。

### P1：IR 正式版本化

需要增加：

- `schema_version`；
- 各想定类型的判别字段；
- 跨字段时序和物理范围约束；
- 旧版本迁移器；
- 组件兼容版本。

### P1：失败注入与恢复测试

至少覆盖：

- Ollama 不可达/模型不存在/生成超时；
- JSON 无法修复；
- 组件 include 缺失；
- `mission.exe` 返回非零；
- `.evt` 缺失、截断或不满足效果；
- 并发运行产物隔离；
- 磁盘空间不足和路径错误。

## 中期优化

### P2：混合 RAG

当前知识量很小，目录加权关键词检索更容易解释。当真实手册、组件和 demo 达到较大规模后，再引入本地 embedding，并用固定查询集评估 Recall@K、来源正确率和无答案拒绝率。不要因为“向量数据库听起来高级”就提前增加复杂度。

### P2：批量统计与可视化

在第 22 步基础上增加：

- 更大重复数；
- 置信区间；
- 参数灵敏度和交互效应；
- CSV/Parquet 数据集；
- 自动图表和实验对比报告。

### P2：安全与发布流程

包括数据分级、模型来源与许可证、离线网络验证、访问控制、审计留存、组件变更审批和人工签字责任。

## 不应优先做的事情

- 在第二类想定尚未打通前继续增加更多 Agent 框架。
- 为三个知识片段部署复杂向量数据库。
- 把 UI 美化当作工程能力。
- 仅凭更大的模型参数量取消验证器和人工复核。
- 直接让模型自由生成 AFSIM txt。

专家成长的主线应始终是：真实建模问题、可验证假设、可靠执行链路和可解释证据。
