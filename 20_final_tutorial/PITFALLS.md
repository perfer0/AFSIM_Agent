# 容易踩坑的地方与规避方式

| 坑 | 表现 | 根因 | 规避方式 |
|---|---|---|---|
| 模型仍写到 C 盘 | D 盘目录为空或 C 盘快速增长 | 只给当前终端设置变量，已运行的 Ollama 服务未继承 | 设置 `OLLAMA_MODELS` 后重启 Ollama，再用 API/`ollama list` 验证 |
| 把磁盘容量当显存 | 模型能下载但运行极慢 | 权重超过显存后部分卸载到系统内存 | 用 `nvidia-smi` 实测；8GB 优先 7B 量化模型，14B 先做速度评测 |
| 规则兜底制造假成功 | 报告通过，但不是模型生成 | `--fallback-rules` 隐藏模型/API失败 | 教学演示可兜底；工程验收禁止兜底并检查 `draft_provider` |
| 只测一个固定提示词 | 看起来 100% 稳定，换参数就失败 | 模型记住示例结构，测试没有覆盖需求变化 | 固定多样本集，至少测参数忠实度、多航点、多动作和非法输入 |
| `mission.exe=0` 就宣布正确 | 仿真可运行但任务意图错误 | 运行成功只证明部分语法和引用有效 | 增加语义断言、事件结果分析和人工业务复核 |
| 有探测就宣布任务成功 | 少量偶然探测被当成整体有效 | 没有把任务目标转换成覆盖率、首次发现时间和航迹指标 | 为每类任务建立独立效果 profile，并通过重复实验验证 |
| 用单次随机运行下结论 | 换一个随机种子结果就变化 | 把随机样本当作总体规律 | 保存随机种子，做重复实验并报告分布和置信区间 |
| 模型直接写 AFSIM 文本 | 语法、关键字和 include 经常幻觉 | 自由文本缺少结构约束 | 模型写 JSON IR，确定性代码渲染 AFSIM |
| RAG 内容越多越好 | 提示词变长，模型忽略关键约束 | 低相关知识污染上下文 | 小块知识、来源元数据、按想定类型过滤、Top-K 评测 |
| 把 RAG 当校验器 | 错误组件仍被接受 | 文档只能提示，不能强制 | 组件白名单和规则校验必须写在代码中 |
| 示例覆盖用户参数 | 输出总是 60000 ft/20 min | 提示词示例权重大于用户需求 | 明确优先级、Schema 约束，并用参数忠实度样例回归 |
| 多任务覆盖产物 | 不同请求共用同一个 build 文件 | 输出目录写死 | 每个运行使用独立 `--build-dir` 和 `--scenario-output` |
| 相对路径偶发失败 | 在不同目录启动时找不到 include/请求 | 依赖当前工作目录 | 程序内部使用 `Path(__file__)` 和绝对路径，报告记录 cwd |
| 本机代理影响 localhost | Ollama API 超时或走代理 | 系统代理接管 `127.0.0.1` | 本地 API 使用 `ProxyHandler({})` 或配置 `NO_PROXY` |
| 中文显示乱码 | README 出现乱码但文件未必损坏 | PowerShell 编码和终端代码页不一致 | 明确 `-Encoding UTF8`，源文件统一 UTF-8 |
| 温度为 0 就认为完全确定 | 同一输入仍可能有变化 | 运行库、模型版本和硬件实现会变化 | 固定模型 tag/版本、保存配置和回归报告，不依赖单次结果 |
| 结构化解码长时间不返回 | 小模型在复杂 Schema 下持续尝试生成 | 模型能力不足或 Schema 正则约束过重 | 每个样例设置硬超时；记录 timeout；简化 Schema 后仍须保留代码校验 |
| Git 仓库塞入模型 | push 巨慢或失败 | 模型 blob 数 GB | `ollama/models/` 永久加入 `.gitignore`，模型单独交付 |
| 离线系统仍发生外联 | 数据安全边界不清 | 安装、遥测、代理或其他工具联网 | 生产环境做防火墙/网络隔离验证，记录依赖与模型来源 |
| 提示词注入知识库 | RAG 文档诱导 Agent 越权 | 把文档内容当指令 | 标注知识是不可信数据，工具参数和组件白名单由代码控制 |
| 生成参数不符合物理/业务口径 | 结构正确但结果失真 | 只有语法校验，没有领域约束 | 为高度、速度、探测距离、时间顺序建立范围和关系规则 |
| 能力范围说得过大 | EOIR demo 被描述为通用 AFSIM Agent | 验收项没有按场景类型拆分 | 每类能力单独维护组件、数据集、实跑和人工签字证据 |
| 旧报告仍显示通过 | 代码或模型升级后继续引用历史绿色报告 | 报告没有绑定当前模型与配置 | 报告记录模型、配置和来源哈希；最终汇总检查当前生产模型是否匹配 |

## 失败定位顺序

1. `doctor_report.json`：路径、服务和模型是否正常。
2. `agent_trace.json`：失败发生在检索、生成、校验、修复还是运行。
3. `scenario.json`：模型是否保留用户参数和组件 ID。
4. `main_generated.txt` 与 `scenario_generated.txt`：确定性渲染是否正确。
5. `mission_stderr.log`：AFSIM 语法、include 或运行错误。
6. `case_result.json`：具体哪个语义断言失败。

不要先反复改提示词。先判断问题属于模型、知识、Schema、业务规则、渲染器还是 AFSIM 组件，再改对应层。
